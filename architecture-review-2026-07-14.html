Split the API client monolith into domain modules

Strong

frontend/src/api/client.ts

Before

After

Problem

One module with 520 lines — interface (20+ exported functions + types) is nearly as large as the implementation. Every page imports the entire surface.

Solution

Split into domain modules under api/: auth.ts, content.ts, digest.ts, etc. Each exports only its domain's functions + response types. The request() core stays deep, imported by all.

Wins

  • Locality: change one domain, touch one file
  • Leverage: request() stays a single seam
  • Interface shrinks per caller
  • Tests import only what they exercise

Problem

The curate_digest handler in wiring.go manually constructs and sequences two services. The pipeline logic (order, error handling, partial results) is invisible — embedded in handler code.

Solution

Define a Pipeline interface in the worker package: Run(ctx, userID) (*Result, error). Move the discovery→categorize sequence into a DigestPipeline struct. Wiring only registers pipelines; runner dispatches through the seam.

Wins

  • Leverage: new pipelines implement the same interface
  • Locality: pipeline logic sits in one module
  • Testable through the Pipeline seam
  • Runner stops knowing about job payloads

Seam the event hub behind an interface

Strong

internal/adapters/events/hub.go + internal/adapters/http/events_handler.go + internal/adapters/events/listener.go

Before

                  flowchart LR
                    SSE["events_handler.go\nhandler"]
                    LISTENER["listener.go\npg LISTEN"]
                    WORKER["worker NotifyFunc"]
                    HUB["events.Hub\n{ struct }"]
                    SSE -->|direct dep on *Hub| HUB
                    LISTENER -->|direct dep on *Hub| HUB
                    WORKER -->|direct dep on *Hub| HUB
                    style HUB fill:#fef2f2,stroke:#dc2626

After

                  flowchart LR
                    SSE["events_handler.go"]
                    LISTENER["listener.go"]
                    WORKER["worker"]
                    IFACE["EventBus interface\nNotifyUser()\nRegister()\nUnregister()"]
                    HUB["Hub implements\nEventBus"]
                    SSE --> IFACE
                    LISTENER --> IFACE
                    WORKER --> IFACE
                    IFACE -.->|satisfied by| HUB
                    style IFACE stroke-dasharray:4 4,stroke:#475569
                    style HUB fill:#0f172a,color:#fff,stroke:#0f172a

Problem

SSE handler, Postgres listener, and worker NOTIFY all depend on the concrete *Hub struct. No seam exists to substitute a different event transport (e.g., Redis pub/sub, NATS) without touching three consumers.

Solution

Define an EventBus interface: NotifyUser(userID, type, data), Register(userID) chan Event, Unregister(userID, chan). Consumers depend on the interface; the Hub becomes one adapter.

Wins

  • One adapter now, but the seam is real
  • Test SSE handler with a mock EventBus
  • Swap transport without touching consumers
  • Interface documents expected behaviour

Create a living domain glossary (CONTEXT.md)

Strong

CONTEXT.md (root) — currently absent

Before

After

Problem

No living glossary. Domain knowledge (what's a "Digest"? How does it differ from an "Edition"?) is scattered across 46 ADRs, Go type definitions, i18n keys, and team memory files. New contributors and AI tools lack a single authoritative reference.

Solution

Create CONTEXT.md at the repo root: one section per domain concept, defining its role, lifecycle, and relationships. Keep it short — a definition per concept, not design docs. Update it when ADRs introduce new terms.

Wins

  • Leverage: one glossary serves all contributors
  • Locality: concept definitions are not scattered
  • AI tools navigate domain via CONTEXT.md
  • Future ADRs can reference terms, not define them
ADR awareness: 46 ADRs already exist — CONTEXT.md would extract the domain vocabulary they introduced (0032 → "Interest", 0029 → "Edition", 0030 → "Compose"). No ADR conflicts; this is a presentation layer for decisions already made.

                    DIG["EnqueueDigestJob(\n  jobsRepo, usagesRepo,\n  plans) — duplicate\n+ conflict check"]

                    style ENQ fill:#fef2f2,stroke:#dc2626
                    style DIG fill:#fef2f2,stroke:#dc2626

After

                  flowchart LR
                    ENQ["enqueueJob(config)\nwhere config includes:\n  - conflict check fn (optional)\n  - job type\n  - plan check fn"]

                    ROUTE2["/api/digest/run"]
                    ENQ --> ROUTE1
                    ENQ --> ROUTE2
                    style ENQ fill:#0f172a,color:#fff,stroke:#0f172a

Problem

enqueueJob takes 6 positional params and returns a handler. EnqueueDigestJob duplicates the same logic with an extra conflict check. The interface (6 params + a special case function) is nearly as complex as the implementation (create job, return 202).

Solution

Replace positional params with a JobConfig struct. Make the conflict check an optional function field. Inline EnqueueDigestJob as a prebuilt config.

Wins

  • Interface shrinks: 1 struct param instead of 6
  • Locality: conflict logic lives in config, not a separate function
  • New job types configure, not fork

Top recommendation

Start with: Split the API client monolith into domain modules

The API client is the most frequently touched file in the frontend — every page, every feature, every test depends on it. Its 520 lines make it the widest interface in the frontend, and changes to one domain (e.g., newsletters) require parsing code for all other domains. Splitting it delivers immediate locality gains, shrinks import surfaces per page, and establishes a pattern that's easy to apply incrementally: extract one module at a time, no big-bang refactor needed.

Runner-up: Factor shared repository infrastructure — eight repositories duplicating the same ~5 lines of boilerplate each. Low effort, visible structural improvement.
